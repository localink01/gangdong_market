---

## 1. 문서 목적

이 문서는 로컬마켓의 디자인 의사결정을 일관되게 유지하기 위한 단일 기준 문서다.

핵심 목적:
1. 동일한 문제를 같은 방식으로 해결한다.
2. 화면별 스타일 편차를 줄인다.
3. 웹/앱 채널 정책을 UI 규칙으로 일관되게 반영한다.

---

## 2. 디자인 원칙

1. 관계 중심 인터페이스
- 지도보다 관계(멤버십, 단골, 알림)를 우선 노출한다.

2. 단계형 탐색
- 지자체 -> 상권 -> 점포 흐름이 한눈에 보이도록 설계한다.

3. 실행 우선
- 정보량보다 다음 행동(가입, 등록, 참여, 사용)을 먼저 보이게 한다.

4. 소프트 iOS 감성
- 라이트 테마, 반투명 레이어, 둥근 형태를 기본 미학으로 유지한다.

5. 정책 우선 UI
- 앱 모드에서 G-00 복귀 미노출 등 서비스 정책을 UI에서 강제한다.
- G-00은 지자체 선택 전용, M-00은 상단바 없는 지역 포털로 고정한다.

---

## 3. 브랜드 톤

키워드:
- 따뜻한 지역성
- 신뢰감 있는 행정 협력 이미지
- 친근하지만 가벼워 보이지 않는 톤

문장 톤:
- 짧고 명확한 행동 유도형 문장
- 과장된 마케팅 문구보다 구체적 혜택 중심 문장

---

## 4. 채널별 UI 규칙 (필수)

1. 웹 모드
- G-00 접근 가능
- 지역 전환 UI 노출 가능

2. 앱(WebView) 모드
- 시작 화면은 M-00
- G-00 진입 버튼/메뉴 미노출
- 홈 액션은 항상 M-00 복귀
- 타 지자체 접근 시 앱 바인딩 지자체 M-00으로 복귀

3. 검수 체크
- 신규 컴포넌트 추가 시 web/app 모드 분기 여부를 반드시 확인

---

## 5. 문서 구조

1. tokens.md
- 색상, 타이포, 간격, 라운드, 그림자, 모션 토큰

2. components.md
- 공통 컴포넌트 규격과 상태 정의

3. patterns.md
- 화면 패턴, 사용자 여정 패턴, 금지 패턴

4. changelog.md
- 규칙 변경 이력

---

## 6. 사용 규칙

1. 새로운 UI를 만들기 전에
- README -> tokens -> components -> patterns 순서로 확인한다.

2. 기존 규칙과 충돌할 때
- 임의 예외를 만들지 말고 changelog에 근거를 남긴 뒤 합의한다.

3. PR 기준
- 디자인 변경 PR에는 영향 범위(컴포넌트/화면/채널)를 명시한다.
- 스크린샷 또는 짧은 시연 영상을 첨부한다.

---

## 7. 즉시 적용 체크리스트

1. 색상/타이포/간격이 토큰을 따르는가?
2. 컴포넌트 상태(기본/활성/비활성/로딩/에러)가 정의되어 있는가?
3. 앱 모드에서 G-00 복귀 UI가 노출되지 않는가?
4. CTA 우선순위가 화면 목적과 일치하는가?
5. 접근성 기준(대비/터치 영역/레이블)을 만족하는가?

---

## 8. 용어 규칙

1. 화면 코드
- G-00, M-00, MK-00, ST-00, U-00, O-00 체계를 유지한다.

2. 표기
- 사용자 노출은 브랜딩명(강동 hero 등)
- 식별/권한/URL은 slug 기준

---

## 9. 운영 원칙

1. 이 문서는 디자인 시스템의 정본이다.
2. 세부 정책 변경 시 반드시 changelog를 먼저 갱신한다.
3. 월 1회 디자인 일관성 리뷰에서 누적 예외를 정리한다.
