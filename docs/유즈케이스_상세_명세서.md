## 1. 문서 목적

본 문서는 구현 착수를 위한 유즈케이스 상세 명세를 정의한다.
전체 유즈케이스 목차는 [유즈케이스_카탈로그](./유즈케이스_카탈로그.md)에서 관리하고, 본 문서는 우선순위가 높은 핵심 유즈케이스의 권한/상태전이/예외/이벤트/API를 상세화하는 실행 명세로 사용한다.

---

## 2. 적용 원칙

1. 용어는 `서비스_용어_규칙_SSOT_맵`을 정본으로 따른다.
2. 채널/URL 규칙은 `로컬마켓_IA_URL_설계안`을 따른다.
3. 권한은 role + scope 기준으로 판정한다.
4. 구현 보류 범위는 `[DEFERRED_MVP]` 태그로 명시한다.

---

## 3. 상태값 공통 정의 (캠페인/공식 콘텐츠)

1. draft: 작성 중
2. in_review: 심사 요청
3. approved_for_region: 지역 포털 노출 승인
4. published: 실제 노출 중
5. archived: 종료/보관
6. rejected: 반려

## 4. UC-08 상권 공식 콘텐츠 작성

### 4-1. 개요
- 유즈케이스 ID: UC-08
- 이름: 상권 공식 콘텐츠 작성
- 목적: 상인회/운영사가 상권 공식 콘텐츠를 작성하여 상권 메인 후보를 생성한다.

### 4-2. Actor
1. 주 액터: 상인회 담당자 (`merchant_council_admin`)
2. 보조 액터: 운영사 관리자 (`region_operator`)

### 4-3. 사전 조건
1. `/ops/**` 로그인 완료
2. `market_id` 스코프 권한 보유
3. 대상 상권 상태가 active

### 4-4. 입력 데이터
1. title (필수)
2. body (필수)
3. content_type (`market_notice`, `market_event`, `market_coupon`)
4. exposure_scope (`market_only`, `region_candidate`)
5. publish_start_at (필수)
6. publish_end_at (필수)
7. event_start_date (`market_event`일 때 필수)
8. event_end_date (`market_event`일 때 필수)
9. event_start_time (선택)
10. event_end_time (선택)
11. location_label (`market_event`일 때 필수)

### 4-5. 정상 흐름
1. 사용자가 상권 운영 화면(O-00-2)에서 콘텐츠 작성 폼을 연다.
2. 필수 항목을 입력하고 `저장`을 누른다.
3. 시스템은 `draft` 상태로 콘텐츠를 생성한다.
4. 사용자가 `심사 요청`을 누르면 상태를 `in_review`로 변경한다.
5. 시스템은 심사 큐에 등록하고 감사 로그를 남긴다.

### 4-6. 예외 흐름
1. 권한 없음
- 조건: role/scope 불일치
- 결과: 403 반환, 화면에 권한 없음 메시지 노출

2. 검증 실패
- 조건: title/body 미입력, publish 기간 역전, event_start_date/event_end_date 누락, event 기간 역전
- 결과: 400 반환, 필드 에러 표시

3. 중복 요청
- 조건: 동일 콘텐츠에 심사 요청 연속 제출
- 결과: 마지막 요청만 유효 처리, 중복 요청은 no-op

### 4-7. 상태 전이
1. none -> draft
2. draft -> in_review

### 4-8. 데이터/이벤트
- 데이터 생성:
1. official_content
2. audit_log

- 이벤트:
1. official_content_created
2. official_content_submitted_for_review

### 4-9. API 매핑 (초안)
1. POST `/ops/regions/{regionSlug}/markets/{marketSlug}/official-contents`
2. POST `/ops/regions/{regionSlug}/markets/{marketSlug}/official-contents/{contentId}/submit-review`

### 4-10. 완료 기준 (AC)
1. 상인회 담당자가 draft 작성 가능
2. 심사 요청 시 in_review 전이
3. 권한 없는 계정은 작성/요청 불가
4. 작성/요청 이벤트 로그 적재
5. `market_event`는 시작일/종료일 없이는 저장 불가
6. 시작일이 종료일보다 늦은 이벤트는 저장 불가

## 5. UC-09 지역 포털 노출 심사/승인

### 5-1. 개요
- 유즈케이스 ID: UC-09
- 이름: 지역 포털 노출 심사/승인
- 목적: 상권에서 올라온 공식 콘텐츠를 지역 포털 노출 가능한 상태로 승인/반려한다.

### 5-2. Actor
1. 주 액터: 지자체 운영사 (`region_operator`)
2. 옵션 액터: 구청 담당자 (`district_admin`, read/review only)

### 5-3. 사전 조건
1. `region_id` 스코프 권한 보유
2. 콘텐츠 상태가 `in_review`
3. 콘텐츠 `exposure_scope`가 `region_candidate`

### 5-4. 정상 흐름
1. 운영사가 지역 운영 화면(O-00-1)의 심사 큐를 연다.
2. 후보 콘텐츠 상세를 확인한다.
3. 승인 시 상태를 `approved_for_region`으로 전환한다.
4. 반려 시 상태를 `rejected`로 전환하고 사유를 저장한다.
5. 시스템은 심사 이력과 감사 로그를 기록한다.

### 5-5. 예외 흐름
1. 이미 처리된 콘텐츠
- 조건: 상태가 in_review가 아님
- 결과: 409 반환

2. 노출 범위 불일치
- 조건: market_only 콘텐츠를 region 승인 시도
- 결과: 400 반환

### 5-6. 상태 전이
1. in_review -> approved_for_region
2. in_review -> rejected

### 5-7. 데이터/이벤트
- 데이터 갱신:
1. official_content.status
2. review_history
3. audit_log

- 이벤트:
1. official_content_approved_for_region
2. official_content_rejected

### 5-8. API 매핑 (초안)
1. POST `/ops/regions/{regionSlug}/official-contents/{contentId}/approve`
2. POST `/ops/regions/{regionSlug}/official-contents/{contentId}/reject`

### 5-9. 완료 기준 (AC)
1. 운영사만 승인/반려 가능
2. 승인/반려 상태 전이 정확
3. 반려 사유 저장/조회 가능
4. 심사 이벤트 로그 적재

## 6. UC-10 포털/상권 메인 배치 게시

### 6-1. 개요
- 유즈케이스 ID: UC-10
- 이름: 포털/상권 메인 배치 게시
- 목적: 승인된 콘텐츠를 M-00 또는 MK-00 메인 슬롯에 배치하여 사용자에게 노출한다.

### 6-2. Actor
1. M-00 배치: 지자체 운영사 (`region_operator`)
2. MK-00 배치: 상인회 담당자 (`merchant_council_admin`) 또는 운영사

### 6-3. 사전 조건
1. 배치 대상 콘텐츠 상태가 `approved_for_region` 또는 `draft/in_review` 승인 완료
2. 배치 슬롯 정책 충족 (동시 노출 제한)
3. 노출 기간(`publish_start_at`, `publish_end_at`) 유효

### 6-4. 정상 흐름
1. 사용자가 배치 화면에서 노출 슬롯(M-00 hero, MK-00 featured)을 선택한다.
2. 콘텐츠를 슬롯에 연결한다.
3. 시스템은 노출 기간과 우선순위를 검증한다.
4. 검증 통과 시 상태를 `published`로 전환하고 실제 화면에 반영한다.
5. 종료 시 상태를 `archived`로 전환한다.

### 6-5. 예외 흐름
1. 슬롯 충돌
- 조건: 동일 슬롯에 동시 게시 정책 위반
- 결과: 409 반환, 우선순위 조정 유도

2. 기간 오류
- 조건: 종료일이 시작일보다 과거
- 결과: 400 반환

3. 권한 오류
- 조건: 상인회가 M-00 슬롯 배치 시도
- 결과: 403 반환

### 6-6. 상태 전이
1. approved_for_region -> published
2. published -> archived

### 6-7. 데이터/이벤트
- 데이터:
1. placement_slot
2. placement_history
3. official_content.status

- 이벤트:
1. content_placed_on_region_portal
2. content_placed_on_market_main
3. content_unpublished

### 6-8. API 매핑 (초안)
1. POST `/ops/regions/{regionSlug}/placements`
2. POST `/ops/regions/{regionSlug}/placements/{placementId}/unpublish`

### 6-9. 완료 기준 (AC)
1. M-00 슬롯은 region_operator만 게시 가능
2. MK-00 슬롯은 market 권한으로 게시 가능
3. 노출 기간 종료 시 자동 해제 가능
4. 게시/해제 이벤트 로그 적재

## 7. UC-302 커뮤니티 글쓰기

### 7-1. 개요
- 유즈케이스 ID: UC-302
- 이름: 커뮤니티 글쓰기
- 목적: 멤버가 상권 커뮤니티에 글을 작성해 지역 소통을 활성화한다.

### 7-2. Actor
1. 주 액터: 상권 멤버 (membership.active)

### 7-3. 사전 조건
1. 사용자 로그인 완료
2. 대상 상권 멤버십 active
3. 상권 컨텍스트(regionSlug, marketSlug) 유효

### 7-4. 입력 데이터
1. body (필수)
2. category (필수: 가게추천, 쇼핑팁, 생활정보, 이벤트, 공지)
3. attachments (선택)

### 7-5. 정상 흐름
1. 멤버가 MK-00 커뮤니티 탭에서 글쓰기 액션을 누른다.
2. 본문/카테고리를 입력 후 등록한다.
3. 시스템은 post를 생성하고 피드 상단에 노출한다.
4. 시스템은 community_post_created 이벤트를 기록한다.

### 7-6. 예외 흐름
1. 멤버 아님
- 결과: 403 또는 멤버십 가입 유도

2. 본문 누락
- 결과: 400 반환, 필드 에러 표시

### 7-7. 데이터/이벤트
- 데이터: community_post
- 이벤트: community_post_created

### 7-8. API 매핑 (초안)
1. POST /regions/{regionSlug}/markets/{marketSlug}/community-posts

### 7-9. 완료 기준 (AC)
1. 멤버만 작성 가능
2. 작성 직후 피드에서 확인 가능
3. 작성 이벤트 로그 저장

## 8. UC-403 쿠폰 수령

### 8-1. 개요
- 유즈케이스 ID: UC-403
- 이름: 쿠폰 수령
- 목적: 사용자가 받을 수 있는 쿠폰을 내 쿠폰함으로 발급받는다.

### 8-2. Actor
1. 주 액터: 멤버 또는 단골

### 8-3. 사전 조건
1. 쿠폰 상태가 발급 가능
2. 대상 쿠폰 조건(member, dangol, event)이 충족
3. 동일 쿠폰 중복 발급 정책 검증 통과

### 8-4. 입력 데이터
1. coupon_id (필수)
2. user_id (서버 식별)
3. issue_context (선택: benefits, event_calendar, store_detail)

### 8-5. 정상 흐름
1. 사용자가 U-00-3 받을 수 있는 쿠폰에서 받기 버튼을 누른다.
2. 시스템은 자격 조건을 검증한다.
3. 통과 시 coupon_issue를 생성한다.
4. 해당 쿠폰이 내 쿠폰함(owned)으로 이동한다.

### 8-6. 예외 흐름
1. 자격 미충족
- 결과: 403 반환, 멤버십/단골 유도

2. 이미 발급됨
- 결과: 409 반환 또는 이미 받음 상태로 표시

### 8-7. 데이터/이벤트
- 데이터: coupon_issue
- 이벤트: coupon_issued

### 8-8. API 매핑 (초안)
1. POST /benefits/coupons/{couponId}/issue

### 8-9. 완료 기준 (AC)
1. 조건 충족 시 즉시 발급
2. 발급 후 내 쿠폰 탭에서 확인 가능
3. 중복 발급 차단

### 9-3. 사전 조건

2. 유효기간 내
3. 사용 가능 대상 가게 일치

### 9-4. 정상 흐름

2. 가게 담당자가 코드/QR을 검증한다.

4. 사용자의 쿠폰함에서 사용 완료로 표시한다.

- 결과: 400 반환, 사용 불가 표시

2. 가게 불일치
- 결과: 403 반환

3. 중복 사용 시도
- 결과: 409 반환

### 9-6. 데이터/이벤트
- 데이터: coupon_redemption, coupon_issue.status
- 이벤트: coupon_redeemed

### 9-7. API 매핑 (초안)
1. POST /ops/stores/{storeSlug}/coupon-redemptions/verify

### 9-8. 완료 기준 (AC)
1. 유효 쿠폰만 소진 가능
2. 사용 직후 상태 업데이트
3. 중복 사용 차단

## 11. UC-805 지자체 공식 캠페인 업로드

### 11-1. 개요
- 유즈케이스 ID: UC-805
- 이름: 지자체 공식 캠페인 업로드
- 목적: 지자체 운영사가 M-00 상단에 노출될 공식 캠페인을 등록한다.
