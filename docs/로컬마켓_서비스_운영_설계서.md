## 1. 문서 개요

### 1-1. 문서 목적
본 문서는 로컬마켓 서비스의 화면 구조, 유즈케이스, 사용자 권한, ph-cms 기준 데이터 적재 정책을 통합 정의한다.

### 1-2. 문서 범위
- 시민 사용자 앱/웹 화면
- 상인회/가게/구청 운영자 화면
- 기능별 권한(RBAC)
- IA 계층 및 URL 컨텍스트 정합
- 도메인 데이터 모델 및 이벤트 로그
- KPI 집계용 데이터 흐름

### 1-3. 문서 비범위
- 상세 UI 비주얼 가이드(색상/타이포 토큰)
- 인프라 운영 세부 스펙(클라우드 리소스 단위)
- 회계/정산/결제 정책

### 1-4. 현재 구현 범위 고정 (MVP)
- 스탬프투어(`stamp_tour`, `stamp_event`, `tour_progress`)는 후속 고도화 기능으로 분류한다.
- 본 문서에서 스탬프투어 관련 항목(UC-06, 데이터 모델, KPI)은 설계 보존용이며, 당장 구현 대상이 아니다.

### 1-5. 보류 범위 태그 규칙
- 공통 태그: `[DEFERRED_MVP]`
- 사용 목적: 백엔드/프론트 구현 티켓에서 당장 미구현 범위를 빠르게 필터링하기 위함
- 적용 위치: 유즈케이스, 데이터 모델, 이벤트, KPI, 운영 루틴

## 2. 서비스 원칙 및 용어 정의

### 2-1. 서비스 원칙
1. 지도 중심이 아닌 관계 중심 서비스
2. 상점가 멤버십 + 가게 단골 관계의 이중 구조
3. 운영자가 직접 운영 가능한 단순 플로우 우선
4. 데이터 기반 성과관리 가능한 구조 우선

### 2-2. 핵심 용어
- 전체 포털(Global Portal): 모든 지자체 진입점
- 지자체(Region): 강동구, 공주시, 익산시 등 운영 단위
- 상점가(Market): 서비스 운영 단위
- 멤버십(Membership): 사용자와 상점가의 관계
- 단골(Dangol): 사용자와 개별 가게의 관계
- 멤버 피드(Community Feed): 상점가 멤버 전용 피드
- 단골 소식(Store News): 가게 단골 전용 피드
- 운영자(Operator): 상인회/가게/구청/지자체 운영사/플랫폼 운영사 역할 사용자

### 2-3. 알림 도메인 경계 (혼선 방지)
본 절을 알림 정책의 정본(SSOT)으로 사용한다.

1. 멤버 알림은 상권(Market) 단위 메시지다.
2. 단골 알림은 가게(Store) 단위 메시지다.
3. 상인회/상권 운영자는 멤버 알림만 발송한다.
4. 가게 운영자는 단골 알림만 발송한다.
5. 단골 알림은 반드시 `store_id`를 가져야 하며, 시장 단위 일괄 발송으로 대체하지 않는다.
6. 시장 단위 공지는 단골 알림이 아니라 멤버 알림으로 분류한다.

## 4. 권한 체계 개요 (RBAC)

### 4-1. 권한 모델
- Role 기반 접근 제어(RBAC)
- Scope: region_id, market_id, store_id
- Action: read, write, publish, send, approve, export

### 4-2. 권한 기본 원칙
1. 최소 권한 원칙
2. 지자체 단위 데이터 분리
3. 상점가 단위 데이터 분리
4. 가게 담당자 권한은 본인 store_id로 제한
5. 구청은 정책/성과 조회 중심, 운영 변경은 제한
6. 운영사 권한은 계약된 region_id 범위로 제한

## 5. IA 계층 요약

1. G-00 전체 포털
2. M-00 지자체 포털
3. MK-00 상권 홈
4. ST-00 가게
5. U-00 개인 영역
6. O-00 운영자 영역

URL 상세 규칙은 별도 문서 `로컬마켓_IA_URL_설계안`을 기준으로 한다.

### 5-1. 채널 운영 정책 (필수)

1. `G-00 전체 포털`은 웹 사이트 전용 진입점으로 운영한다.
2. 지역 앱(WebView)은 `M-00 지자체 포털`부터 시작한다.
3. 앱 모드에서 `M-00` 이후 `G-00`으로 복귀하는 버튼/메뉴는 제공하지 않는다.
4. 앱 모드에서 `/` 또는 `/regions` 진입 시 앱 바인딩 지자체 `M-00`으로 리다이렉트한다.
5. 앱 모드에서 타 지자체 경로 진입 시 기본 정책은 현재 앱 바인딩 지자체 `M-00` 복귀다.
6. `G-00`은 지자체 선택 전용 화면이며, 앱/웹 공통으로 하단 네비게이션 독을 두지 않는다.
7. `M-00`은 상단바 없이 지자체 브랜드와 상권 목록 중심으로 노출한다.
8. 웹 채널에서 `/` 또는 `/regions`는 `G-00`으로 렌더링한다.

운영 체크포인트:
- QA/UAT는 웹 모드와 앱 모드를 분리해 검수한다.
- KPI는 웹/앱을 분리 집계한다.

## 10. ph-cms 데이터 아키텍처 개요

### 10-1. 데이터 계층
1. 마스터 데이터: region, market, store, category, operator_org, operator
2. 관계 데이터: membership, dangol_relation
3. 콘텐츠 데이터: community_post, store_news, campaign
4. 트랜잭션 데이터: coupon_issue/redemption, stamp_event
5. 커뮤니케이션 데이터: notification, delivery, read_receipt
6. 분석 데이터: analytics_event, kpi_snapshot

### 10-2. 멀티 상점가 원칙
- 모든 핵심 데이터는 region_id를 가진다.
- 모든 핵심 데이터는 market_id를 가진다.
- store 데이터는 market_id + store_id 복합 스코프를 가진다.
- 운영자 계정은 role + scope(지자체/시장/가게)를 반드시 가진다.

## 20. IA/URL 정합 적용 기준

1. 사용자 IA 계층은 `전체 포털 -> 지자체 -> 상권 -> 가게`를 따른다.
2. 운영자 IA 계층은 `지자체 운영 -> 상권 운영 -> 가게 운영`을 따른다.
3. URL 컨텍스트는 `regionSlug/marketSlug/storeSlug` path 기반으로 유지한다.
4. 화면 상태는 query 파라미터로 유지한다.
5. 상세 URL 규칙은 `로컬마켓_IA_URL_설계안` 문서를 정본으로 삼는다.
