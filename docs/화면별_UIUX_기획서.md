# 로컬마켓 — 화면별 UI/UX 기획서 (IA 리라이팅)

작성일자 : 2026-05-26  
문서 버전 : v1.0 (IA/URL/서비스기획 통합 반영)

> 본 문서는 최신 IA 및 URL 규칙, 멀티테넌트 서비스 전략을 기준으로  
> 로컬마켓의 사용자 화면과 운영자 화면을 재정의한 통합 UI/UX 기획서다.

> MVP 구현 범위 메모: 스탬프투어 UI/기능은 후속 고도화 항목이며, 당장 구현하지 않는다.

---

## 0. 리라이팅 기준

1. 탐색 계층은 지자체 -> 상권 -> 가게 순서로 통일한다.
2. 지자체는 브랜딩명으로 노출한다.
- 강동 hero (regionSlug: gangdong-gu)
- 공주왕도심 (regionSlug: gongju-si)
- 익산문화거리 (regionSlug: iksan-si)
3. URL 컨텍스트는 slug 기반, 화면 텍스트는 브랜딩명 기반으로 분리한다.
4. G-00 전체 포털은 지자체 선택 전용 화면으로 정의한다.
5. 루트 진입은 채널 기준으로 분리한다. 웹은 G-00으로 시작하고, 지역 앱(WebView)은 M-00으로 시작한다.

## 1. 서비스 화면 체계 (최신)

| 트랙 | 코드 | 화면명 | 핵심 역할 |
|---|---|---|---|
| Global | G-00 | 전체 포털 | 지자체 선택 허브 |
| Region | M-00 | 지자체 포털 | 지자체 단위 상권 목록/캠페인 |
| Market | MK-00 | 상권 포털 | 멤버십, 커뮤니티, 가게 탐색 |
| Store | ST-00 | 가게 상세 | 단골 등록, 가게 혜택, 단골 소식 진입 |
| User | U-00 | 개인 영역 | 멤버십, 단골, 쿠폰, 알림, 설정 관리 |
| Ops | O-00 | 운영자 영역 | 지자체/상권/가게/시스템 운영 |

## 2. 전역 진입 정책

### 2-0. 채널 정책

1. G-00 전체 포털은 웹 사이트에서만 노출한다.
2. 지역별 앱은 WebView 래퍼 앱으로 배포한다.
3. 지역별 앱의 시작 화면은 M-00 지자체 포털이다.
4. 즉, 앱은 지역별 다중 패키지, 서비스 코드는 단일 코드베이스를 유지한다.
5. G-00은 지자체 선택 전용이며 상권 홍보형 히어로를 두지 않는다.
6. M-00은 상단바 없이 지역 브랜드와 상권 카드를 중심으로 구성한다.

예시 시작 URL:
- 강동 앱: `/regions/gangdong-gu`
- 공주 앱: `/regions/gongju-si`
- 익산 앱: `/regions/iksan-si`

채널별 네비게이션 규칙:
- 웹: G-00 접근 가능, 지역 전환 UI 노출 가능
- 지역 앱(WebView): G-00 접근 비허용, 지역 전환 UI 기본 비노출
- 지역 앱(WebView)에서 M-00 이후 G-00으로 돌아가는 버튼/메뉴는 제공하지 않음
- 지역 앱(WebView) 홈 버튼은 항상 해당 지자체 M-00으로 복귀

### 2-3. 하단 네비게이션 기준

사용자 하단 네비게이션은 아래 매핑을 기준으로 고정한다.

1. 포털: M-00 지자체 포털
2. 상점가: MK-00 상권 포털
3. 혜택: 상권/가게 혜택 통합 허브(U-00-3)
4. 마이: 마이페이지/내 활동(U-00-1)

규칙:
- G-00은 지자체 선택을 위한 초기 진입 화면이다.
- M-00 진입 이후 하단 네비게이션으로 G-00 복귀 경로를 제공하지 않는다.

### 2-1. 루트 진입 정책

1. 웹 채널
- `/` 또는 `/regions` 진입 시 G-00 전체 포털 노출
- 사용자에게 지자체 선택을 먼저 요구

2. 지역 앱(WebView) 채널
- 앱 시작 URL은 `/regions/{appRegionSlug}`를 사용한다.
- 앱 모드에서 `/` 또는 `/regions` 진입 시 `/regions/{appRegionSlug}`로 리다이렉트한다.

3. 1지역 운영 단계(MVP) 예외
- 운영 배포 정책상 필요하면 앱 시작 URL을 `/regions/gangdong-gu`로 고정할 수 있다.
- 이 경우에도 웹 채널의 G-00 진입 정책은 유지한다.

### 2-2. 브랜딩 표기 정책

| 구분 | 내부 식별자 | 사용자 노출명 |
|---|---|---|
| 강동구 권역 | gangdong-gu | 강동 hero |
| 공주시 권역 | gongju-si | 공주왕도심 |
| 익산시 권역 | iksan-si | 익산문화거리 |

원칙:
- 네비게이션, 카드, 헤더, 배지는 브랜딩명 우선
- 운영 로그, 권한 체크, API 호출은 내부 식별자 기준

## 3. 사용자 핵심 여정

```text
웹 진입
  -> G-00 전체 포털 (지자체 찾기)
  -> M-00 지자체 포털 (상권 선택)
  -> MK-00 상권 포털
      -> 멤버십 등록
      -> 커뮤니티/혜택 탐색
      -> ST-00 가게 상세
          -> 단골 등록
          -> 단골 소식 수신
  -> U-영역에서 내 활동/알림/쿠폰 관리
```

```text
지역 앱(WebView) 진입
  -> M-00 지자체 포털 (앱 바인딩 지역)
  -> MK-00 상권 포털
      -> ST-00 가게 상세
  -> 홈 버튼 탭 시 M-00 복귀
  -> G-00 이동 버튼은 노출되지 않음
```

보조 진입:
- QR/NFC: 해당 상권 또는 가게로 바로 진입
- 푸시 알림: 관련 컨텍스트 화면으로 딥링크

## 4. 화면별 상세 설계

### G-00. 전체 포털 (다지역 허브)

화면 목적:
- 사용자가 먼저 지자체를 선택하게 하는 글로벌 허브
- 계층 탐색의 1단계, 선택 전용 진입점

진입 경로:
- `/`
- `/regions`
- 딥링크 컨텍스트 없는 최초 실행

채널 제한:
- 웹 사이트에서만 노출
- 지역 앱(WebView)에서는 렌더링하지 않음

컴포넌트:
1. 지자체 선택 리스트
2. 선택된 지자체 요약 카드

상태 정책:
- 비로그인: 지자체 카드 + 공개 정보 노출
- 로그인/멤버십 1개: 내 상권 바로가기 우선
- 로그인/멤버십 다수: 최근 방문 상권 우선 정렬

전환:
- 지자체 카드 탭 -> M-00 지자체 포털

레이아웃 규칙:
- 히어로 배너를 두지 않는다.
- 하단 네비게이션을 두지 않는다.
- 지자체 선택 후 M-00에서만 지역성 콘텐츠를 확장한다.

---

### M-00. 지자체 포털

화면 목적:
- 지자체 단위 허브로 상권 목록, 지자체 캠페인, 공지 제공

URL 예시:
- `/regions/gangdong-gu` (강동 hero)
- `/regions/gongju-si` (공주왕도심)
- `/regions/iksan-si` (익산문화거리)

앱 네비게이션 기준:
- 지역 앱(WebView)의 기본 홈 화면
- 앱 로고/홈 탭/메인 복귀 액션은 모두 M-00으로 이동
- M-00 상단바는 노출하지 않음
- M-00 이후 G-00으로 이동하는 버튼/메뉴는 노출하지 않음
- 하단 "포털" 탭의 목적지는 G-00이 아니라 M-00으로 고정

컴포넌트:
1. 지자체 브랜딩 히어로: 브랜딩명, 소개 문구, 참여 지표
2. 참여 상권 목록
3. 지자체 통합 캠페인
4. 지자체 공지/리포트(공개 범위)
5. 상권 검색/필터(카테고리, 운영 상태)

핵심 UX 원칙:
- 상권 카드는 소개보다 현재 참여 가능한 액션(이벤트, 쿠폰, 투어)을 우선 노출
- 준비 중 상권은 회색 처리 + 오픈 알림 신청

전환:
- 상권 카드 탭 -> MK-00 상권 포털

### M-00-1. 이벤트 전용 화면

화면 목적:
- 지자체 내 상권/가게 이벤트를 날짜 기준으로 탐색할 수 있는 공개 캘린더 허브
- 사용자가 `언제 어디에 가면 어떤 이벤트가 있는지`를 먼저 파악하도록 돕는 방문 계획 화면

핵심 정의:
1. 이 화면은 쿠폰함이 아니다.
2. 이 화면의 기본 정보 구조는 `캘린더`다.
3. 지자체는 캘린더를 통해 지역 전체 일정을 정리해주고, 상권/가게는 각 날짜에 맞는 이벤트를 공급한다.
4. 신뢰의 출처는 `지자체 공식 운영`, 실행 콘텐츠의 출처는 `상권 공식` 또는 `가게`라는 구조를 분명히 드러낸다.

URL 예시:
- `/regions/gangdong-gu?panel=events`
- `/regions/gangdong-gu?panel=events&month=2026-05`
- `/regions/gangdong-gu?panel=events&date=2026-05-29`

대상 사용자:
1. 지역 주민
2. 인근 방문 예정자
3. 상권 탐색 단계의 비회원 사용자

콘텐츠 운영 원칙:
1. 모든 이벤트는 `날짜`에 귀속되어야 하며, 날짜 없는 콘텐츠는 M-00 이벤트 캘린더에 올리지 않는다.
2. 모든 카드에는 `지자체 공식`, `상권 공식`, `가게` 중 작성 주체를 명시한다.
3. 같은 날짜에 여러 이벤트가 있을 경우, 지자체는 날짜 셀에서 상권 단위로 묶어 보여준다.
4. 개별 가게 이벤트는 날짜 셀과 상세 패널에서는 노출하되, 메인 캘린더에서는 상권 맥락 안에서 우선 정렬한다.

권장 정보 구조:
1. 헤더
- 화면명: `이번 달 지역 이벤트 캘린더`
- 보조 문구: `날짜에 맞춰 상권 행사와 가게 이벤트를 찾아보세요`
- 월 이동: 이전달 / 현재월 / 다음달
- 상태 필터: 전체 / 오늘 / 이번주 / 진행중 / 곧 종료

2. 월간 캘린더 그리드
- 월 단위 5~6주 캘린더를 기본으로 사용한다.
- 날짜 셀에는 이벤트 개수, 대표 상권명, 카테고리 점 표시를 노출한다.
- 날짜 셀 선택 시 하단 상세 패널 또는 다음 섹션이 갱신된다.
- 오늘 날짜와 선택 날짜는 명확히 구분한다.

3. 선택 날짜 상세 패널
- 선택한 날짜의 이벤트를 시간순으로 나열한다.
- 각 항목에는 시간, 상권명 또는 가게명, 이벤트 제목, 위치, 참여 조건을 표시한다.
- 같은 상권 이벤트가 여러 개인 경우 상권 그룹으로 묶는다.
- 메인 CTA: `상권 보기`
- 보조 CTA: `가게 보기`, `길찾기`

4. 이번주 추천 일정
- 캘린더 아래에 `이번주 가볼 만한 일정`을 카드로 재정렬한다.
- 예: 금요일 저녁, 토요일 가족 방문, 일요일 마감 임박 이벤트
- 사용자가 캘린더를 본 뒤 바로 방문 계획을 세우도록 돕는다.

5. 참여 상권 리스트
- 상권명, 위치, 이번달 이벤트 수, 가장 가까운 이벤트 날짜
- CTA: `상권 보기`
- 보조 CTA: `이번 일정 보기`

6. 공공 신뢰/운영 정보 영역
- 주최/주관 표기
- 문의처
- 유의사항
- 사업 목적 또는 지역 상권 활성화 메시지

실제 와이어프레임(모바일 기준):

```text
┌──────────────────────────────┐
│ 이번 달 지역 이벤트 캘린더        │
│ 날짜에 맞춰 상권/가게 일정을 찾기 │
├──────────────────────────────┤
│ [이전달]   2026년 5월   [다음달] │
│ [전체] [오늘] [이번주] [진행중]   │
├──────────────────────────────┤
│ 일  월  화  수  목  금  토       │
│ 26 27 28 29* 30 31  1        │
│    2개 고덕 1개 GO  2개 ...    │
│  3  4  5  6  7  8  9          │
│ ...                          │
├──────────────────────────────┤
│ 5.29(금) 선택 날짜 상세          │
│ 15:00 고덕 베이커리 1+1        │
│ 19:00 GO PUB 응원전           │
│ [상권 보기] [가게 보기]         │
├──────────────────────────────┤
│ 이번주 가볼 만한 일정            │
│ 금요일 밤 추천 / 토요일 가족 추천 │
├──────────────────────────────┤
│ 참여 상권 일정                  │
│ 고덕2동 · 이번 달 이벤트 5개     │
└──────────────────────────────┘
```

캘린더 셀 구성 규칙:
1. 좌상단: 날짜 숫자
2. 우상단: 카테고리 점 1개 또는 복수 점 요약
3. 중단 1행: 해당 날짜 이벤트 개수 (`2개 일정`)
4. 하단 1행: 대표 상권명 또는 대표 가게명 1줄
5. 선택 셀: 진한 배경 + 상세 패널 연동
6. 오늘 셀: 테두리 강조
7. 일정 없음 셀: 빈 상태를 명확히 표시하되 클릭 가능 유지

셀 상태별 표시 원칙:
1. 하루짜리 이벤트는 시작일과 종료일이 같아도 1일 셀에 노출한다.
2. 기간형 이벤트는 시작일부터 종료일까지 모든 날짜 셀에 이어서 노출한다.
3. 같은 날짜에 가게 이벤트가 많아도 셀 안에서는 최대 1개 이름만 노출하고, 나머지는 개수로 요약한다.
4. 셀에서 해결하는 것은 `날짜 선택`까지이며, 상세 설명은 하단 패널에서 처리한다.

상세 패널 카드 구성:
1. 시간
2. 이벤트 제목
3. 주체 배지(지자체 공식 / 상권 공식 / 가게)
4. 상권명 또는 가게명
5. 장소
6. 참여 조건
7. CTA (`상권 보기`, `가게 보기`)

업로드 데이터가 캘린더에 반영되는 방식:
1. 이벤트 시작일과 종료일은 캘린더 노출 범위를 결정하는 기준값이다.
2. 시작일/종료일이 없는 이벤트는 저장 또는 게시할 수 없다.
3. 시작일이 종료일보다 늦으면 업로드를 차단한다.
4. 시간 미입력 시 캘린더 상세 패널에서는 `시간 미정`으로 표기할 수 있으나, 날짜 자체는 필수다.

카드 설계 원칙:
1. M-00 이벤트 화면의 1순위 정보는 `언제 가야 하는가`다.
2. 2순위 정보는 `어디로 가야 하는가`다.
3. 3순위 정보는 `왜 가야 하는가`다.
4. 할인율, 쿠폰 문구는 보조 정보로만 쓴다.
5. 과도한 프로모션 톤보다 공공 일정 안내와 지역 큐레이션 신뢰를 우선한다.

대표 카드 필드:
1. 날짜 또는 시간
2. 이벤트 제목
3. 주체 배지(지자체 공식 / 상권 공식 / 가게)
4. 상권명 또는 가게명
5. 위치 정보
6. CTA (`상권 보기`, `가게 보기`)

전환 규칙:
1. 날짜 셀 탭 -> 해당 날짜 상세 패널 갱신
2. 상세 패널의 상권 이벤트 탭 -> MK-00 상권 포털 또는 MK-00 이벤트 필터 화면으로 이동
3. 상세 패널의 가게 이벤트 탭 -> ST-00 가게 상세 또는 가게 이벤트 상세로 이동
4. 쿠폰 수령 CTA -> U-00-3 혜택 허브로 이동
5. 멤버 전용 참여 액션은 M-00에서 완료하지 않고 MK-00 또는 U-00로 넘긴다.

레이아웃 원칙:
1. 상단은 월간 캘린더, 중단은 선택 날짜 상세, 하단은 상권 탐색 확장을 담당한다.
2. 첫 화면에서 오늘 포함 주간 이벤트 흐름이 한눈에 보여야 한다.
3. 모바일에서는 `월간 캘린더 -> 날짜 상세 카드` 2단 구조를 유지한다.
4. 태블릿/웹에서는 좌측 캘린더, 우측 상세 패널 2열 구성이 가능해야 한다.

금지 사항:
1. M-00 이벤트 화면을 멤버 전용 잠금 화면으로 만들지 않는다.
2. 쿠폰 목록만 나열하는 구조로 만들지 않는다.
3. 날짜 없는 카드 나열형 피드로 대체하지 않는다.
4. 가게 단위 상세 이벤트를 M-00에서 과도하게 길게 풀지 않는다.
5. 이벤트 참여 완료/인증 같은 개인 행동 플로우를 M-00에서 처리하지 않는다.

MVP 구현 우선순위:
1. 월간 캘린더 그리드
2. 날짜 선택 상세 패널
3. 오늘/이번주/진행중 필터
4. 상권 상세 이동 CTA

후속 고도화:
1. 반복 일정 지원
2. 지도 기반 주변 상권 이벤트 보기
3. 취향/시간대 기반 개인화 추천

3. UC-403 쿠폰 수령
4. UC-404 쿠폰 사용 검증/소진
5. UC-205 멤버십/단골 해제(내 활동 연계)

1. 내 쿠폰 탭(사용 가능/완료/만료)

4. 쿠폰 출처 배지(상권 공식 / 가게 / 이벤트)

1. `내 쿠폰`은 이미 발급받은 쿠폰만 보여준다.
2. `받을 수 있는 쿠폰`은 아직 미발급이지만 조건을 만족하면 받을 수 있는 쿠폰을 보여준다.

4. `단골 가게 전용`은 별도 배지로만 구분하고 쿠폰함 전체를 단골 가게로 제한하지 않는다.
5. 스탬프투어 보상 쿠폰은 MVP에서는 미노출한다.

## 8. 단계별 론칭 권장안

1. 1차(강동 hero 단일)
- 웹은 G-00 유지, 앱은 M-00 고정으로 채널 분리 적용
- M-00/MK-00/ST-00 및 멤버십/단골 핵심 플로우 우선

2. 2차(다지역 확장)
- G-00 지자체 허브 활성화
- 브랜딩명/slug 매핑 운영

3. 3차(운영 고도화)
- 지자체 통합 캠페인
- 추천 코스/스탬프 개인화
- 운영자 KPI 리포트 자동화

## 9. 최종 정리

1. 화면 탐색의 1단계는 지자체 선택이다.
2. 브랜딩명 노출과 내부 slug 식별을 분리해야 확장 시 혼선이 없다.
3. G-00은 다지역 허브, M-00은 지자체 운영 허브, MK-00은 전환 허브로 역할을 분리한다.
4. 핵심 전환은 멤버십 등록과 단골 등록이며, UI는 항상 이 두 행동을 짧은 경로로 유도해야 한다.
5. 지역 앱(WebView)에서는 M-00 이후 G-00으로 돌아가는 네비게이션을 제공하지 않는다.
